# SOLID Принципы

## S — Single Responsibility
- Один класс = одна причина для изменения
- Use Case содержит ровно одну бизнес-операцию
- Если метод делает 2 вещи — разбить на два метода

## O — Open/Closed
- Расширение через новые классы, НЕ изменение существующих
- Использовать абстракции для расширения поведения

## L — Liskov Substitution
- Дочерний класс полностью заменяет родительский
- Не ужесточать предусловия, не ослаблять постусловия
- Не выбрасывать исключения которых нет в родителе

## I — Interface Segregation
- Маленькие специфичные интерфейсы вместо одного большого
- Клиент не зависит от методов которые не использует

## D — Dependency Inversion
- Зависеть от абстракций, не от конкретных реализаций
- Внедрение зависимостей через конструктор (Constructor Injection)
- ЗАПРЕЩЕНО: new ConcreteRepository() внутри Use Case

## Применение в Spring Boot (текущий проект)
- DI через @Autowired или конструктор (предпочтительно конструктор)
- Интерфейсы сервисов в пакете service/, реализации в service/impl/
- Не использовать @Autowired на полях — только на конструкторе
- Каждый @Service реализует один интерфейс с минимальным набором методов

## Чеклист перед сдачей кода
- [ ] Каждый класс имеет одну ответственность
- [ ] Все внешние зависимости инжектируются через конструктор
- [ ] Есть интерфейс в domain/interfaces/ для каждого репозитория/сервиса
- [ ] Use Cases тестируются без фреймворка (только моки интерфейсов)
- [ ] Нет нарушений направления зависимостей между слоями
