Запусти полный цикл разработки для задачи: $ARGUMENTS

━━━ ФАЗА 0: ТРИАЖ + PRE-FLIGHT (оркестратор, без агентов) ━━━

Перед запуском агентов сделай ДВА шага САМОСТОЯТЕЛЬНО:

### 0.1. Триаж задачи — определи минимально нужный набор агентов

Классифицируй $ARGUMENTS по типу:

| Тип задачи | Агенты | Признаки |
|---|---|---|
| **UI-only** | designer + frontend-dev + tester | Только верстка/стили/UX, без изменения API |
| **Backend-only** | architect + backend-dev + tester | БД, миграции, рефакторинг сервисов, race conditions, scheduled jobs |
| **Bug fix (точечный)** | один dev + tester | Опечатка, валидация, фикс одного компонента |
| **Post-mortem / research** | architect ║ tester (параллельно, БЕЗ Фаз 2/3) | Анализ инцидента, без реализации |
| **Full-stack feature** | все 5 агентов | Новая фича с API + UI |

Запиши решение явно: «Тип задачи: <тип>. Запускаются агенты: <список>. Пропускаются: <список> потому что <причина>».

### 0.2. Pre-flight exploration — собери общий контекст ОДИН раз

Сделай Glob/Grep/Read по теме задачи:
- Найди существующие файлы релевантные задаче (модели, сервисы, контроллеры, компоненты)
- Прочитай ключевые куски кода (НЕ копируй полные файлы — только то что нужно)
- Найди существующие тесты по теме

Сохрани результат в текстовый блок «PRE-FLIGHT CONTEXT» который будет инжектирован в prompts всех агентов:

```
PRE-FLIGHT CONTEXT (собрано оркестратором):
- Релевантные файлы: <список с путями>
- Ключевые куски: <цитаты с файл:строка>
- Существующие тесты: <список>
- Связанные ADR/feature memory: <список>
```

Это сократит время агентов: они не будут делать те же Glob/Grep заново.

━━━ ФАЗА 1: ПРОЕКТИРОВАНИЕ (параллельно) ━━━

Запусти ОДНОВРЕМЕННО (если триаж требует обоих):

### 1a. designer (если задача затрагивает UI)
- РЕЖИМ 1 (init): если нет docs/design-system.md — создай его
- РЕЖИМ 2 (review): создай UI Spec для задачи: $ARGUMENTS
- **Лимит: UI Spec не более 1500 слов**, без дублирующих секций
- Передай ему PRE-FLIGHT CONTEXT в prompt
- Выход: docs/ui-specs/<feature-name>.md

### 1b. architect (если задача затрагивает backend / архитектуру)
- Создай ADR на основе $ARGUMENTS
- **Лимит: ADR не более 1500 слов**, конкретно с путями файлов
- Если есть @Async/@Scheduled/счётчики/idempotency — ОБЯЗАТЕЛЬНО секция «Concurrency model» (см. concurrency-patterns.md)
- Если меняется БД — секция «DB schema changes»
- Передай ему PRE-FLIGHT CONTEXT в prompt (НЕ нужно делать grep заново)
- Выход: docs/adr/ADR-XXX-<feature>.md

Designer и architect работают **независимо** — они оба смотрят на $ARGUMENTS, не зависят друг от друга. Если возникнет конфликт между UI Spec и ADR — он всплывёт на точке одобрения, и можно будет дополнить.

━━━ ТОЧКА ОДОБРЕНИЯ ━━━

СТОП. Перед реализацией покажи пользователю сводку:
- UI Spec от designer (ключевые решения, не весь документ)
- ADR от architect (компоненты, API контракт, concurrency model если есть)
- Список файлов которые будут созданы/изменены

Спроси: «Одобряете план? Есть замечания?»
НЕ переходи к Фазе 2 пока пользователь явно не одобрит.
Агенты в Фазе 2 и 3 НЕ спрашивают одобрение повторно — они выполняют одобренный план.

**Если триаж = «post-mortem / research»** — стоп здесь. Пользователь может попросить запустить Фазы 2-3 отдельно если захочет реализацию.

━━━ ФАЗА 2a: ТЕСТЫ (TDD — tester ПЕРВЫМ) ━━━

Используй агента tester:
- Прочитай ADR + UI Spec + PRE-FLIGHT CONTEXT
- Напиши **полноценные** тесты (backend unit + frontend unit + E2E) — не заглушки
- Если в ADR есть «Concurrency model» — обязательно concurrency integration-тесты (см. testing.md, раздел Concurrency)
- Тесты НЕ запускай (упадут — кода ещё нет)
- Лимит: тесты должны быть исчерпывающими, но без дублирования

Выход: тестовые файлы в правильных директориях (по structure из testing.md)

━━━ ФАЗА 2b: РЕАЛИЗАЦИЯ (параллельно) ━━━

Запусти ОДНОВРЕМЕННО двух агентов:

### Агент backend-developer
- Прочитай ADR от architect + PRE-FLIGHT CONTEXT (уже есть готовый контекст, не нужно делать grep заново)
- Прочитай тесты от tester — реализация должна их удовлетворить
- НЕ редактируй тесты — реализация подгоняется к тестам
- Реализуй сервисы, репозитории, контроллеры
- Если ADR требует SQL изменений — сформируй SQL команды (Flyway отключён, не пиши миграции)
- Создай MapStruct маппер для новых DTO
- Пройди чек-листы SOLID + Concurrency + Error Handling (см. backend-developer.md)
- Создай API_CONTRACT.md если есть новые endpoints

### Агент frontend-developer
- Прочитай UI Spec + ADR + API_CONTRACT.md (если уже создан) + PRE-FLIGHT CONTEXT
- Прочитай frontend тесты от tester — реализация должна их удовлетворить
- НЕ редактируй тесты
- Реализуй Angular компоненты и страницы по UI Spec
- Подключись к API через сервис в core/services/
- Реализуй все состояния: loading/empty/error/success
- Пройди чек-лист дизайн-системы (нет хардкода)

━━━ ФАЗА 3: ПРОВЕРКА (последовательно) ━━━

Используй агента tester:

### 3a. Запуск backend тестов (foreground — короткие)
- Запусти `./mvnw test` (учитывай: нужна JDK 17, см. MEMORY.md)
- Зафиксируй passed/failed/skipped

### 3b. Запуск E2E тестов (background — длинные)
- Запусти `cd ui-ng && npx playwright test` через `run_in_background: true`
- НЕ жди завершения — продолжай к 3c
- Уведомление о результате прилетит позже

### 3c. Архитектурный + Concurrency ревью (параллельно с 3b)
- Контроллер не вызывает репозиторий напрямую: `grep -r "Repository" backend/.../controller/`
- Concurrency grep по красным флагам (см. tester.md):
  - `@Async + @Scheduled` без distributed lock
  - `entity.setX(getX() + n)` без `@Version`/atomic UPDATE
  - `if (existsBy...) save();` без UNIQUE-constraint
- UI ревью: нет хардкода вне SCSS-переменных, все состояния реализованы

━━━ ФИНАЛЬНЫЙ ОТЧЁТ ━━━

Выведи (НЕ ждать E2E если они в background):
- Триаж: какие агенты были запущены / пропущены
- Реализованный backend (endpoints, сервисы, SQL команды если есть)
- Реализованный frontend (компоненты, страницы)
- API_CONTRACT.md создан (если применимо)
- Соответствие дизайн-системе
- Результаты backend тестов (passed/failed)
- E2E тесты: «запущены в background, результат прилетит уведомлением»
- Архитектурные/concurrency нарушения (если есть)
- Замечания

Если пользователь хочет дождаться E2E — он может попросить отдельно.

━━━ ЖЁСТКИЕ ПРАВИЛА ━━━

- ОБЯЗАТЕЛЬНО Фаза 0 (триаж + pre-flight) ДО запуска любого агента
- ОБЯЗАТЕЛЬНО передавать PRE-FLIGHT CONTEXT в prompt каждого агента — иначе они дублируют exploration
- ОБЯЗАТЕЛЬНО точка одобрения после Фазы 1, ДО Фазы 2 (см. memory: одобрение плана)
- ОБЯЗАТЕЛЬНО TDD-порядок в Фазе 2: tester (тесты) → developers (реализация) → tester (запуск)
- ЗАПРЕЩЕНО запускать агентов которые триаж посчитал ненужными (трата времени)
- ЗАПРЕЩЕНО создавать UI Spec для backend-only задач (race conditions, миграции, рефакторинг)
- ЗАПРЕЩЕНО создавать Flyway миграции — Flyway отключён, формируй SQL команды для ручного применения
- ЗАПРЕЩЕНО блокировать финальный отчёт ожиданием E2E тестов — они в background
